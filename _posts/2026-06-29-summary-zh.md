## [OpenAI Codex 排除敏感文件的议题仍未解决](https://github.com/openai/codex/issues/2847) ⭐️ 8.0/10

截至 2025 年 8 月，要求 OpenAI Codex 内置排除敏感文件方式的 GitHub 议题（#2847）仍处于开放状态，尚未有官方实现。社区成员持续提出沙盒和文件权限等变通方案。 像 Codex 这样的 AI 编码代理可能无意中访问并上传敏感数据，带来安全风险。缺乏内置排除功能意味着用户必须依赖外部沙盒，而这可能并非所有人都能轻松实现。 维护者已关闭该议题，转而支持基于 Rust 的重写（codex-rs），但截至最新更新，该功能尚未实现。社区解决方案包括使用 chmod 限制文件访问或将 Codex 运行在容器内。

hackernews · pikseladam · 6月28日 12:27 · [社区讨论](https://news.ycombinator.com/item?id=48706714)

**背景**: OpenAI Codex 是一个 AI 编码代理，能够读取和写入用户工作区中的文件。由于它可以运行 shell 命令并检查文件内容，如果敏感文件可访问，它可能无意中暴露 API 密钥或 SSH 密钥等敏感信息。沙盒将代理的执行环境隔离，限制其可访问的内容，而文件权限（如 chmod）则在操作系统级别拒绝读取访问。

## [AltGr 键处理导致波兰字母 S 消失](https://aresluna.org/the-curious-case-of-the-disappearing-polish-s/) ⭐️ 8.0/10

一篇 2015 年的文章揭示，Web 应用常因将 AltGr 组合键误认为 Ctrl+Alt 快捷键而屏蔽波兰字符，影响 Windows 用户。文章还解释了 Unicode 标准化和键盘布局特性如何导致此问题。 此问题至今仍有意义，它影响了波兰语用户对 Web 应用的可操作性，并凸显了跨平台输入兼容性的普遍挑战。开发者常忽视区域键盘特性，导致合法输入被意外拦截。 文章指出，波兰键盘上键入'ś'等字母需用 AltGr 键，而 Windows 将其模拟为 Ctrl+Alt。绑定 Ctrl+Alt 快捷键的 Web 应用因此会拦截这些字符。此外，Unicode 标准化分解（NFD）会将大多数波兰字母分解为基本字母加组合变音标记，但'ł'保持不变，这给 SQLite 等数据库的文本处理增加了复杂性。

hackernews · colinprince · 6月28日 12:44 · [社区讨论](https://news.ycombinator.com/item?id=48706814)

**背景**: AltGr（Alternate Graphic）键是许多非美式键盘上用于输入带变音字母或特殊符号的修饰键。在 Windows 中，AltGr 被视为 Ctrl+Alt，可能与应用程序快捷键冲突。Unicode 标准化确保等价字符序列表示一致，但不同的分解行为可能导致不一致。波兰语使用带变音的拉丁字母，因此易受此类输入处理问题影响。

这项研究克服了哥德尔机的证明瓶颈，使该理论重新焕发活力，可能加速递归自我改进（RSI）和超级智能（ASI）的到来，正如 Anthropic 的 Jack Clark 预测 2028 年实现的可能性为 60%。 实验中，RQGM 将代码生成通过率从 69.9%提升至 71.7%且节省了 token，论文接收率从 21.8%提升至 40.5%。系统采用受控效用进化机制：每个 epoch 内固定评估器，只有在锚点数据上统计优于旧评估器时才允许更换考官。

rss · 36氪 - 24小时热榜 · 6月28日 07:33

**背景**: 哥德尔机是 Jürgen Schmidhuber 于 2003 年提出的理论自我改进系统，要求每次修改前数学证明其有益，导致计算不可行。后来的达尔文哥德尔机（DGM）和赫胥黎哥德尔机（HGM）放弃证明改用进化，但评估器固定不变。RQGM 引入红皇后假说（生物进化中捕食者与猎物协同进化），让评估器与选手共同进化，打破静态评价的局限。
